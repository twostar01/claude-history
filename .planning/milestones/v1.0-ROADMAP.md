**Status:** ✅ SHIPPED 2026-05-06

## Overview

Four phases from nothing to a fully registered, searchable MCP server. Phase 1 was gated on a real export file — schema discovery was the first task and unblocked everything else. Phases 2 and 3 were the heavy lift (FTS5 schema + ingest, then MCP tools). Phase 4 closed the loop with user-scope registration and the README.

## Milestone Summary

**Key Decisions:**

- Transport is stdio (not HTTP daemon) — Claude Code spawns per-session; no Task Scheduler needed
- FTS5 tokenizer: `unicode61 remove_diacritics 2 tokenchars '-_'` — locked before first data; handles snake_case terms
- stdout must NEVER be written in server.py — stderr-only logging is non-negotiable for stdio MCP
- Project association gap: conversations.json has no project field; all conversations stored with project=NULL
- bm25() cannot appear in GROUP BY subqueries in SQLite FTS5 — Python dict aggregation is the correct per-conversation dedup pattern
- INSERT OR IGNORE (not OR REPLACE) — append-only history records must never be overwritten
- User-scope registration uses `uv --directory <abs-path> run server` so it works from any Claude Code session directory

**Issues Resolved:**

- uv PATH not in bash sessions (Windows install to ~/.local/bin) — workaround documented
- `uv run` fails when server.exe is locked by MCP process on Windows — `uv run --no-sync` workaround
- bm25() cannot appear in subqueries — discovered during Phase 3 planning, solved with Python aggregation

**Issues Deferred (v2):**

- SRCH-01: date_from/date_to filter parameters for search_conversations
- SRCH-02: role_filter parameter (search only human or assistant turns)
- EXP-01: export_conversation write-to-file option
- ATTACH-01: PDF attachment indexing (requires third-party parser)

**Technical Debt:**

- `uv run ingest` fails on Windows when MCP server is running (file lock on server.exe). Workaround: use `.venv/Scripts/python.exe -m claude_history.ingest` directly.
